# frago-principle-supreme

分类: 第一性原理（MUST）

## 解决什么问题
agent 在做 frago 相关的设计取舍时（写 feature、改命令、起文档、设计错误信息），缺少一条最高仲裁标准。多个看起来都对的选项之间，需要一条原则来拍板。这条原则就是仲裁标准。

## 第一性原理

> **让 agent 更容易正确地理解如何使用 frago。**

任何 feature、命令、文档、错误信息、API 设计、参数命名、输出格式的取舍，最终都回到这一条。

## 怎么用这条原则

当多个方案都"看起来合理"时，问：

1. **哪个方案让 agent 更不容易误用？**
   - 命令名是否歧义？
   - 参数顺序是否反直觉？
   - 默认行为是否需要 agent 记忆？

2. **哪个方案让 agent 更不容易绕过？**
   - 错误信息是否包含可执行的修正命令？
   - 失败模式是否安静？（安静失败 = agent 学不到教训）

3. **哪个方案让 agent 更不容易猜测？**
   - 输出是否结构化？
   - 文档是否能被 agent 直接索引？
   - 行为是否跨调用一致？

如果某个设计**让 agent 更容易误用、绕过、或猜测，它就是错的**——不论它对人类用户多直观、多优雅。

## 反例

| 设计 | 看起来 | 实际 |
|------|--------|------|
| 默认静默成功，--verbose 才输出 | 简洁 | agent 看不到反馈，下一步盲飞 |
| 错误码 1，没有结构化错误体 | "Unix 风格" | agent 不知道哪儿错了、怎么修 |
| 命令参数支持简写 -h / -help / --help | 灵活 | agent 学到错的简写就一直错下去 |
| 交互式确认 [y/N] | 安全 | auto-approved 模式下卡死 |

每一条都是"对人类直观但坑 agent"的典型。frago 的取舍偏向后者。

## 跟其他原则的关系

这条是**最高准则**。其他 principle（trust / rigor / extend）跟它冲突时，以这条为准。其他 principle 都是这条在不同子问题上的展开。
