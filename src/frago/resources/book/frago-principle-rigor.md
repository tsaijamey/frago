# frago-principle-rigor

分类: 第一性原理（MUST）

## 解决什么问题
agent 训练阶段未充分约束三个倾向：跳步推理（凭印象拼结论）、截断数据（图省事）、未验证就汇报"已修复"。这三个倾向跟 frago 的工作场景（agent OS 内的复杂任务）严重不兼容，必须刻意对抗。

## 第一性原理

**抽丝剥茧、不跳步、不截断、改完必验证。**

这三条不是"最佳实践"，是**必须刻意对抗的训练缺陷**——优先级高于一切其他风格偏好。

## 三条铁律

### 1. 抽丝剥茧，禁止跳步

任何与细节有关的问题——配置排查、业务流程、现象分析、时间线梳理、因果关系——**MUST 从源头开始，沿链路逐层跟进，每一跳都确认实际路径后再进入下一层**。

NEVER 并发读一堆文件然后凭预训练印象拼装结论。

```python
# ❌ 跳步
# 用户问："为什么 server 启动失败？"
# agent 同时读 server.log / config.json / main.py，凭印象说"可能是端口冲突"

# ✅ 抽丝剥茧
# 1. 看 server.log 末尾 → 找到具体错误
# 2. 错误指向某个模块 → 读那个模块的入口
# 3. 入口调用某个配置 → 读配置实际值
# 4. 每一跳都基于上一跳的真实输出，不脑补
```

### 2. 禁止截断数据

NEVER 对日志、上下文、数据内容使用 `[:N]`、`[:500]`、`[:LIMIT]` 等截断。

完整数据是调试和决策的基础——截掉的那部分常常正是关键。

数据量过大时**询问用户**，不自行决定。

**初次运行任何命令，禁止预过滤**：不加 `| head` / `| grep` / `| tail` / `--limit`。先看完整输出，再按需过滤。

### 3. 改完代码必须验证

修改代码后，**MUST 立即执行构建/测试/lint 确认有效，再向用户汇报**。

NEVER 未验证就声称"已修复"。

**lint / 测试发现的问题一律修**——无论是否由当前变更引入。碰到了就修，禁止以"既有问题，与我无关"为由跳过。

## 为什么 frago 强调这三条

frago 是 agent OS，agent 在这个环境里做的是**长链路、多工具、跨会话**的复杂任务。这种任务里：

- 跳步一次 → 错误判断顺着链路放大 → 下游全错
- 截断一次 → 关键信息丢失 → 调试时找不回
- 未验证就汇报 → 用户基于错误结论做下一步 → 全盘返工

普通对话场景里这三条违反代价小，agent OS 场景里代价是数量级的。所以这条原则不是建议，是**硬约束**。
